# Deployment (PROMPT-09 §46–§47)

Pipeline stays the repo's existing one (deploy.yml: build CSS/JS → zola check → zola build → seo:validate → npm test → gh-pages). The temporal gate runs in `npm test` (the unit suite validates generated data invariants) and fully in the local pre-commit `npm run verify` (which runs temporal:validate including the canonical-drift check). Snapshot/change/freshness data is generated locally by `npm run data:build` and committed — CI deploys committed artifacts (audit D11, unchanged by design).

Build commands (§47 mapping): temporal-audit → docs/00 + validator; snapshot-create → temporal:build (no-op aware) / temporal:bootstrap; snapshot-validate + snapshot-diff + changes-validate → temporal:validate; changes-generate → temporal:build; freshness-evaluate → node scripts/dossier/temporal/freshness.mjs [--as-of=…]; revalidation-plan + temporal-routes → temporal:build; temporal-test → npm test; temporal-browser-test → npm run test:responsive; temporal-production-proof → node scripts/dossier/temporal/production-proof.mjs; temporal-all → npm run verify. Running temporal:build twice over unchanged canonical data provably creates no changes (no-op test).
