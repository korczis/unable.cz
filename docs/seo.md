# SEO and content architecture

One rule underpins everything here: **metadata is declared once, in front
matter, and rendered once, in `templates/base.html`.** Every other property in
this document follows from that. If you only read one section, read
[Adding a page](#adding-a-page) and [Anti-patterns](#anti-patterns).

## Why it is built this way

The failure mode this design exists to prevent is drift. The moment a second
template hand-writes its own `<meta property="og:title">`, the site has two
sources of truth, and they diverge silently — a page's title says one thing,
its Open Graph card says another, and nothing fails. That had already begun
here: `dossier.html` once carried a copy of the whole Open Graph block from
`base.html`.

The fix is structural rather than advisory. `base.html` exposes **no block
hooks** for title, description, canonical, Open Graph, or JSON-LD. There is
nothing for a page to override, so a page cannot drift. To change metadata, you
change data.

## Pipeline

```
content/*.md front matter        <- the only place metadata is authored
  + config.toml [extra]          <- defaults for anything a page omits
        |
        v
templates/base.html              <- the only renderer; normalizes and emits
        |
        +--> <title>, description, robots, canonical
        +--> Open Graph + Twitter tags
        +--> JSON-LD (all values json_encode'd)
        |
        v
zola build -> public/            <- sitemap.xml and robots.txt are Zola's
        |
        v
scripts/seo-validate.mjs         <- validates the built artifact + the policy
        |
        v
CI (.github/workflows/deploy.yml) -> gh-pages
```

## Source of truth

| Concern | Source of truth | Rendered by | Validated by |
|---|---|---|---|
| Title, description | page front matter | `base.html` | `E_TITLE_MISSING`, `E_DESC_MISSING`, `E_DUP_*` |
| Canonical URL | `config.toml` `base_url` + Zola's `current_url` | `base.html` | `E_CANONICAL_MISMATCH` |
| Open Graph / Twitter | front matter, defaults from `[extra]` | `base.html` | `W_OG_MISSING`, `E_OG_URL_MISMATCH` |
| Structured data | `extra.seo_type` + native dates | `base.html` | `E_JSONLD_*` |
| Published / modified | Zola-native `date` / `updated` | `base.html`, sitemap `lastmod` | `E_DATE_*` |
| Evidence + review policy | `[extra.evidence]` | `dossier.html` | `E_REVIEW_POLICY_MISSING`, `W_REVIEW_OVERDUE` |
| Sitemap | Zola (`sitemap.xml`) | Zola | `E_SITEMAP_*` |
| robots.txt | Zola (`generate_robots_txt`) | Zola | `E_ROBOTS_*` |

There is exactly one sitemap and one `robots.txt`, both generated by Zola. Do
not hand-maintain either, and do not add a second.

## The metadata schema

Site-wide defaults live in `config.toml` under `[extra]`. A page overrides any
of them by setting the same key in its own `[extra]`; anything it omits falls
back. `base.html` resolves the chain.

### Page keys (`[extra]`)

| Key | Default | Meaning |
|---|---|---|
| `seo_type` | `WebPage` | schema.org type. Must be in the validator's allowlist. |
| `og_type` | `website` | `website` or `article`. `article` also emits `article:*` time tags. |
| `og_image` | `images/og-image.jpg` | Path under `static/`. **Never** an absolute URL — it is resolved through `get_url`. |
| `og_image_alt` | the page title | Alt text for the social card. |
| `index` | `true` | Set `false` to opt out of indexing and the sitemap. |
| `lang`, `locale` | `en`, `en_US` | Language metadata. |

### Native Zola keys

Use these rather than inventing parallel fields:

- `title`, `description` — drive the title tag, meta description, OG and Twitter.
- `date` — publication. Emitted as `datePublished`.
- `updated` — last substantive change. Emitted as `dateModified` **and** the
  sitemap's `<lastmod>`. Bump it when the substance changes, never mechanically
  and never on a build.

### Evidence keys (`[extra.evidence]`)

Required when `seo_type = "Report"`. The validator fails the build without them.

| Key | Meaning |
|---|---|
| `cutoff` | When the underlying evidence was observed. Must not postdate `updated` — evidence cannot come from after the review that used it. |
| `reviewed_at` | When a human last checked the claims against reality. |
| `review_interval_days` | How long that check is good for. |
| `confidence` | `high`, `moderate`, or `low`. |
| `freshness_class` | How the content ages (see below). |
| `sources_consulted` | Count of external sources. `0` is a legitimate value, and is itself a limitation worth stating. |

## Freshness

`freshness_class` records how a page ages, so a reader and a reviewer can tell
a permanent statement from a perishable one:

- `static` — not expected to change.
- `slowly_changing` — drifts over years.
- `periodically_reviewed` — correct as of the last review.
- `event_driven` — changes when something happens elsewhere.
- `historical_snapshot` — deliberately frozen; describes a moment in time.

"Up to date" is a mechanism, not a date in a footer:

1. `[extra.evidence]` declares the review interval.
2. `dossier.html` compares `reviewed_at` against the interval **at build time**
   and renders a visible "overdue for review" notice when it has lapsed. The
   provenance panel always shows last-updated, evidence cutoff, confidence, and
   review status.
3. `seo-validate.mjs` emits `W_REVIEW_OVERDUE` with the actual day count.

The stale notice is computed from `now()`, so a build is deterministic *given a
build date* but not across dates. That is the point: content ages whether or
not anyone rebuilds. `now()` is read only for that comparison — it never feeds
`updated`, which would churn the timestamp on every build and make `lastmod` a
lie.

**Why overdue is a warning, not an error.** An overdue page already announces
its own staleness to every reader. Failing the build would couple every
unrelated change — a typo fix on the homepage — to one page's review clock, and
the usual result of that is someone bumping `reviewed_at` without re-reading
anything, which is worse than a visible notice. Run
`npm run seo:validate:strict` to treat warnings as failures in review.

## Adding a page

1. Create `content/<slug>.md` with `+++` front matter: `title`, `description`,
   and a `template` if it needs a custom layout.
2. Set `[extra] seo_type` if it is not a plain `WebPage`.
3. If it asserts evidence (`seo_type = "Report"`), add the full
   `[extra.evidence]` table.
4. Run `npm run verify`.

Do not add a single `<meta>` tag. If you find yourself wanting to, the schema is
missing a field — add the field.

## Adding or updating a dossier

A dossier is a page with `seo_type = "Report"`. It carries evidentiary weight,
so:

- Separate **observed**, **claimed**, and **assessed**. The existing
  `content/dossier.md` marks each section this way; follow it.
- Every material claim needs a source and a retrieval date. If no source was
  consulted, say so — `sources_consulted = 0` is honest; a fabricated citation
  is not.
- State confidence and say *why* it is what it is.
- When updating: re-check the sources first, then bump `reviewed_at`. Bump
  `updated` **only if the substance changed**. Bumping `reviewed_at` without
  re-reading the sources defeats the entire mechanism and is undetectable by
  any tool here.
- Never publish a claim about a third party that the cited evidence does not
  support. Neutral phrasing does not launder an unsupported claim, and the
  context a page sits in carries implications that its sentences do not.

## Validation

```bash
npm run verify              # build + validate + test  (use this)
npm run seo:validate        # errors fail, warnings report
npm run seo:validate:strict # warnings fail too
npm run seo:validate:json   # machine-readable, for CI
npm test                    # unit + regression tests
zola check                  # Zola's own template/link check
```

The validator runs against `public/` — the built artifact, not the sources —
because generated HTML is what ships and what a crawler sees. A check that
passes on front matter but fails in the output is worthless. The two
exceptions are deliberate: hardcoded-host detection and review policy are
checked at the source, because they are authoring concerns that the output
cannot express.

It parses HTML with regexes. That is sound *only* because the input is our own
deterministic Zola output; do not point it at arbitrary HTML.

### What it enforces

**Errors** (fail the build): missing title or description on an indexable page;
duplicate titles or descriptions; a canonical that is missing, or that
disagrees with the page's own route, or that appears on a non-route; a literal
production host in a template or content attribute; JSON-LD that does not
parse, claims a type outside the allowlist, or disagrees with the canonical;
a `Report` without its evidence declaration; impossible dates (future,
`dateModified` before `datePublished`, evidence cutoff after the review);
broken internal links or missing assets; `javascript:` URLs; a sitemap that
omits an indexable page or lists a `noindex` one; a `robots.txt` that does not
point at the real sitemap; a non-production host in the output; more or fewer
than one `<h1>`.

**Warnings**: an overdue review; an orphaned indexable route; over-long titles
or descriptions; missing Open Graph tags.

There is deliberately **no flag that skips a check.** `--strict` only makes the
gate harsher.

### JSON-LD encoding (do not shortcut this)

Every value interpolated into JSON-LD goes through the `seo::j` macro in
`templates/macros/seo.html`. **`json_encode` alone is not enough.**

`json_encode` escapes quotes, so the JSON stays well-formed — but it leaves the
literal string `</script>` untouched, and an HTML parser ends a script element
at the first `</script>` it sees, whatever the JSON means. A page title of
`</script><script>alert(1)</script>` therefore escapes the block and executes.
This was a live bug here, caught by testing hostile front matter rather than by
reading the template.

`seo::j` additionally rewrites `</` to `<\/`. That is a legal JSON escape for
`/`, so every parser reads it back as a plain slash, while the HTML parser no
longer sees a closing tag. Three tests pin this: that `base.html` contains no
bare `json_encode | safe`, that the macro performs the replacement, and that
the built JSON-LD parses with no raw `</script>` inside it.

Elsewhere — meta tags, attributes — Tera's autoescaping already handles this.
The hazard is specific to `| safe` inside a `<script>` block. Markdown front
matter is not trusted input just because it is Markdown.

### The schema allowlist

`ALLOWED_SCHEMA_TYPES` in `scripts/seo-validate.mjs` is a closed set:
`WebSite`, `WebPage`, `Report`, `Article`, `CollectionPage`. `Review`,
`AggregateRating`, and `ClaimReview` are excluded on purpose and a test pins
that — this site has no reviews or ratings, and structured data that invents
credibility signals is both a lie to readers and a manual-action risk. Adding a
type must be a deliberate, evidenced decision.

## Discoverability of the dossier

The dossier is reachable only from the `UNABLE` link in the hero, plus the
sitemap. That is intentional: low-discoverability, but **not orphaned** — the
validator's `W_ORPHAN` check would fire if the last inbound link were removed,
so it cannot rot into an unreachable page unnoticed. It is not in any
navigation, because there is no navigation.

## The UNABLE link

`templates/index.html`. The word is a real anchor to `@/dossier.md`, resolved
through `get_url` so it follows `base_url` in every environment (in `zola
serve` it points at `127.0.0.1:1111`, which is the resolver proving itself).

It reuses `.hero-emphasis` — the same emphasis step as `PAY` and `CAN` — so it
stays part of the poster rather than becoming a widget. Only the champagne tone
(`#f3e5c0` against white siblings) sets it apart. No underline, no icon, no
tooltip, no JavaScript. Its accessible name contains its visible text
(WCAG 2.5.3), and it takes a visible focus ring.

Regression tests in `scripts/seo-validate.test.mjs` pin all of this, including
that it has not reverted to plain text and that the hero contains exactly one
link.

## Deployment

`.github/workflows/deploy.yml`: install → `zola check` → `zola build` →
`seo:validate` → `npm test` → publish `./public` to `gh-pages`. Validation and
tests gate the deploy; if metadata is broken, nothing ships.

Zola's version is pinned (`zola@0.22.1`) so the build is reproducible.

## Anti-patterns

- **A `<meta>` tag in any template but `base.html`.** A test fails on this.
- **Re-introducing a metadata block hook in `base.html`.** A test fails on this
  too. Blocks are what let pages drift.
- **A literal `unable.cz` in an href, src, or content attribute.** Use
  `get_url` / `current_url`. `config.toml` is the one place the host is named.
- **Bumping `updated` on every build**, which makes `lastmod` meaningless.
- **Bumping `reviewed_at` without re-reading the sources.** No tool can catch
  this. It is the one part of the system that runs on honesty.
- **Adding a schema type to make a page look more credible.**
- **A second sitemap.**
- **Hand-copying front matter into a JSON file** to make some tool happy.
